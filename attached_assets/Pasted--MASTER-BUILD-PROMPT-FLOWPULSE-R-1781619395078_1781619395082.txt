SECTION 0 — CRITICAL EXTERNAL CONSTRAINTS (READ FIRST)
================================================================

0.1  Meta App Review is mandatory for Advanced Access (instagram_manage_messages,
     instagram_manage_comments, pages_messaging). The platform uses a single
     "Parent App" that holds all Advanced Access. End-users onboard via Meta
     Business Extension (MBE) / Embedded Signup — they never submit their
     own App Review. This is the #1 architectural requirement.

0.2  TikTok 2026 policy: only "inbound-only" triggers are permitted. The
     platform MUST NEVER initiate mass comments, auto-follow, or any
     proactive outbox action not triggered by a user interaction. Violating
     this causes permanent API bans.

0.3  The Meta 24-Hour Messaging Window is law. Any message sent >24h after
     the last user interaction without a permitted Message Tag
     (POST_PURCHASE_UPDATE, CONFIRMED_EVENT_UPDATE, etc.) will get the
     platform's App ID flagged. Every outgoing message must pass the
     24-Hour Validator before dispatch.

0.4  Token security: OAuth access_tokens and refresh_tokens must NEVER be
     stored in plain text. All tokens are encrypted at rest using AWS KMS
     (envelope encryption: AES-256-GCM data key, encrypted by KMS master key).

0.5  Data residency: EU webhook data must never physically leave the EU
     (eu-central-1). US data stays in us-east-1. Use AWS Route 53 or
     Cloudflare Workers for geo-routing. PostgreSQL uses Citus geo-partitions.

SECTION 2 — WEBHOOK INGRESSOR (Build second)
================================================================

Build in Node.js (or Go for higher throughput). This is a stateless,
ultra-lightweight service whose ONLY job is:

2.1 INGRESSOR LOGIC (must complete in < 50ms total)

  Step 1: Receive POST /webhook/meta or POST /webhook/tiktok
  Step 2: Validate HMAC signature
    Meta: X-Hub-Signature-256 header, SHA-256 HMAC of raw body with
          app_secret. Reject with 403 if invalid.
    TikTok: validate per TikTok's 2026 webhook signing spec.
  Step 3: Extract delivery_id (Meta) or message_id (TikTok) from payload
  Step 4: Redis SETNX with key = SHA256(delivery_id), TTL = 86400s
    If SETNX returns 0: DUPLICATE — return 200 OK immediately, stop.
  Step 5: Write raw encrypted payload to S3 synchronously:
    Path: s3://flowpulse-webhook-vault/{tenant_id}/{YYYY}/{MM}/{DD}/{HH}/{delivery_id}.json
    Encrypt with AES-256-GCM before upload.
    S3 lifecycle: Standard 14d → Glacier Instant 15-30d → Purge day 31
    (configurable per brand up to 90 days)
  Step 6: Drop validated payload into Amazon SQS queue
  Step 7: Return 200 OK

  Do NOT process business logic here.
  Do NOT call Meta/TikTok APIs here.
  Do NOT open DB connections here.
  Target: handle 50,000 concurrent webhooks in 3 seconds without dropping one.
  Use Provisioned Concurrency (Lambda) or a persistent Node.js cluster.

2.2 SQS CONFIGURATION
  - Visibility timeout: 30 seconds
  - Max receive count: 3
  - Dead Letter Queue (DLQ) attached
  - DLQ retention: 14 days

2.3 LAMBDA WORKER (Tier 2 processor)
  Pulls from SQS at a controlled rate matching DB write capacity.
  Processing pipeline per message:
    a. Check 24-Hour Window (Section 4)
    b. Check circuit breaker state (Section 7)
    c. Check global contact session in Redis (Section 8)
    d. Route to correct flow engine
    e. Execute flow nodes
    f. Call Meta/TikTok outbound APIs with rate-limit token bucket

  Lambda configuration:
    - Reserved concurrency: 100 (prevents DB pool exhaustion)
    - Use RDS Proxy (PgBouncer) — NEVER open direct Postgres connections from Lambda
    - Provisioned concurrency for Ingressor function only

SECTION 3 — TOKEN VAULT & AUTO-REFRESH ENGINE
================================================================

3.1 ENCRYPTION FLOW (AWS KMS envelope encryption)
  Store:
    1. Call KMS GenerateDataKey (AES-256)
    2. Encrypt token with data key (AES-256-GCM, 12-byte IV, 16-byte tag)
    3. Store: encrypted_access_token, encrypted_data_key, IV+tag in DB column
    4. Discard plaintext data key immediately

  Retrieve:
    1. Fetch encrypted_access_token + encrypted_data_key from DB
    2. Call KMS Decrypt to get plaintext data key
    3. Decrypt token in-memory using data key
    4. Use token for API call
    5. Zero-out plaintext token variable immediately after use
    6. Write to token_decryption_audit table

  Key rotation: quarterly via KMS automatic key rotation.
  Every KMS decrypt call logged to AWS CloudTrail.

3.2 AUTO-REFRESH CRON (runs every hour)
  For every connected_account where token_expires_at < NOW() + 48h:
    a. Attempt token refresh using platform refresh endpoint
    b. If success: re-encrypt new token (Section 3.1), update DB,
       emit WebSocket event account.token_refreshed
    c. If fail: set health_status = 'BROKEN', emit WebSocket event
       account.token_broken, send push notification (FCM/APNs) to
       tenant owner, send in-app alert, update Token Health Panel

  Gate: before EVERY outgoing API call, verify token_expires_at > NOW().
  Never allow a flow to execute with an expired token.

SECTION 4 — 24-HOUR MESSAGING WINDOW VALIDATOR
================================================================

4.1 IMPLEMENTATION
  Before any message is dequeued and dispatched:

  last_ts = platform_profiles.last_interaction_at for the recipient
  delta_t = NOW() - last_ts

  If delta_t < 23.5 hours:
    → Allow full marketing message. Proceed normally.

  If 23.5h ≤ delta_t ≤ 24.0h:
    → Strip all marketing text. Send lightweight text-only message.
    → Append permitted Message Tag to API payload.
    → Log as "NEAR_WINDOW_EXPIRY"

  If delta_t > 24.0h AND flow node has a permitted Message Tag:
    → Allow message with that tag. Attach tag to API payload.

  If delta_t > 24.0h AND no permitted tag:
    → BLOCK message entirely.
    → Log "24H_WINDOW_EXPIRED" event to DLQ dashboard.
    → If brand has SMS/email fallback configured: fire fallback.
    → Do NOT send to Meta/TikTok.

  Use the webhook event's own timestamp field — NOT database insertion time.

4.2 SMART DELAY WARNING IN BUILDER
  When a user configures a Smart Delay node in the flow builder, calculate
  if the delay could push any recipient past the 24h window.
  If yes: show a red warning badge on the node + inline suggestion to
  add a permitted Message Tag to the downstream message node.

SECTION 7 — CIRCUIT BREAKER & RATE LIMIT CONTROLLER
================================================================

7.1 CIRCUIT BREAKER (per connected_account)
  Track in Redis: {tenant_id}:{brand_id}:circuit:{account_id}
  State object: { state, tripped_at, failure_count, probe_at }

  Window: rolling 60-second failure rate per account.

  CLOSED state (normal):
    success → reset failure_count
    failure → increment failure_count
    if failure_rate > 15% over 60s → OPEN

  OPEN state (tripped):
    ALL outgoing messages for this account: BLOCKED
    Emit: WebSocket account.circuit_open event
    Send: push notification + in-app alert to tenant owner
    Log: reason, failed messages count, timestamp
    After 5 minutes → HALF_OPEN

  HALF_OPEN state:
    Allow 1 probe message
    If success → CLOSED, reset counters
    If fail → OPEN again with exponential backoff (5m → 10m → 20m)

7.2 RATE LIMIT TOKEN BUCKET (per connected_account, in Redis)
  Sliding window counter tracks messages sent per second and per hour.
  Meta limits: 250 conversations/second (tier-dependent per page)
  TikTok limits: per TikTok 2026 API docs

  At 80% of limit: emit "Rate Limit Approaching" event to UI dashboard
  At 100%: re-enqueue message with delay using SQS DelaySeconds
  Never drop messages due to rate limits — only delay them.

  TikTok sliding window: if outbound actions cross 85% of TikTok's
  standard rate within 60s: shift profile flow into cooling phase,
  log queue retries with exponential backoff.

7.3 DEAD LETTER QUEUE + INTELLIGENT REPLAY
  Messages failing 3 processing attempts → DLQ with error metadata.
  Cron runs every 15 minutes:
    a. Ping health endpoints for failing 3rd-party APIs (Shopify/Meta/TikTok)
    b. On recovery: auto-replay DLQ with exponential backoff
       (1min delay → 4min → 16min)
  Replay flag: add is_replay=true to message — bypasses dedup layer.
  DLQ visible in DLQ Replay Dashboard (Section 14).

SECTION 8 — GLOBAL OMNI-CHANNEL STATE MACHINE
================================================================

8.1 ACTIVE SESSION STORE (Redis)
  Key: {tenant_id}:{brand_id}:session:{unified_contact_id}
  Value: {
    contact_id, platform, flow_id, node_id,
    entered_at, next_action_at, loyalty_score
  }
  TTL: 24 hours (refreshed on every interaction)

8.2 SESSION CHECK BEFORE TRIGGER
  When a new trigger arrives for a contact:
    a. Check if active session exists for this unified_contact_id
    b. If no session: start new flow normally
    c. If session exists on SAME platform: continue existing flow
    d. If session exists on DIFFERENT platform:
       Option A (default): queue new trigger, process after current flow
       Option B (configurable per flow): merge context and continue
    e. This prevents two flows spamming the same user simultaneously
       from Instagram and TikTok at the same time.

SECTION 9 — AI LAYER
================================================================

9.1 INTENT RECOGNITION ENGINE (runs on EVERY inbound message)
  Use Tier 1 model (fast, cheap) for classification.
  Output: intent bucket + confidence score (0.0–1.0)

  Built-in intent buckets:
    BUY_INTENT, REFUND_REQUEST, PRICE_CHECK, GENERAL_QUESTION,
    COMPLAINT, GREETING, UNSUBSCRIBE, LINK_REQUEST, OTHER

  Custom intents: tenants can define additional keyword + semantic labels.
  Intent stored as contact event in attribution_events table.
  Intent used for: flow branching, LLM tier routing, handoff triggers,
  Ghost A/B tracking.

9.2 LLM ROUTER (two-tier)
  Before any LLM call:
    Step 1: Semantic cache check (Section 9.5)
    If cache hit: return cached response, 0 token cost, log as CACHE_HIT

    Step 2: Intent + confidence scoring
    If intent is FAQ/GREETING AND confidence > 0.85:
      → Route to Tier 1 (Gemini 1.5 Flash or GPT-4o-mini)
    If intent is BUY_INTENT, REFUND_REQUEST, or function calling needed:
      → Route to Tier 2 (GPT-4o or Claude Sonnet)
    If confidence < 0.50 and strictness = STRICT:
      → Skip LLM, trigger "Wait for Human" state

  Log: token usage, model, cost per conversation.
  Monthly AI budget cap per tenant: when reached, force Tier 1 only.
  Show estimated monthly AI cost in analytics dashboard.

9.3 RAG KNOWLEDGE BASE
  Chunking: 512 tokens per chunk, 50-token overlap.
  Embeddings: stored in pgvector (or Pinecone/Weaviate per deployment).
  On inbound question:
    a. Embed the question
    b. Retrieve top-3 most similar chunks from brand's KB
    c. Log similarity scores

  Strictness routing:
    STRICT: if max similarity < 0.75 → do NOT answer → "Wait for Human"
            Log as KB gap for tenant review
    BALANCED: if 0.50 ≤ score < 0.75 → answer with confidence disclaimer
    CREATIVE: answer even with low retrieval confidence

  Every retrieval logged to ai_audit_logs (chunk, score, question, answer).

9.4 PERSONA-BASED AI
  Each brand configures:
    - Persona name + tone (friendly/professional/witty)
    - Language
    - Forbidden topics (hard block — never attempt to answer)
    - "If unsure" behavior: hand off / ask clarifying question / offer support
  Persona injected into every LLM system prompt.

9.5 SEMANTIC CACHE (RedisVL)
  On each inbound question:
    a. Compute embedding of question
    b. Query RedisVL for semantically similar cached responses
    c. Similarity threshold: 0.92 → cache hit
    d. Cache TTL: 6 hours (FAQ) / 30 minutes (product/inventory)
    e. Invalidate brand cache on: new KB upload, product price change

9.6 AGENTIC TOOL-USE (function calling, Tier 2 only)
  Available functions:
    get_order_status(email) → Shopify Orders API
    book_appointment(date, service) → Calendly API
    get_product_info(product_id) → TikTok Shop API or Shopify
    get_tiktok_inventory(product_id) → TikTok Shop API (real-time)
    custom_webhook(url, payload) → any external system

  Multi-step reasoning: AI can chain up to 3 tool calls per turn.
  All function call inputs + outputs logged to ai_audit_logs.
  Timeout: 5s per function call. On timeout: send hold message,
  retry in background. Keep 24h window active.

  All 3rd-party credentials (Shopify, Calendly) stored encrypted via
  AWS KMS (same envelope encryption as OAuth tokens). Never plain text.
  Outbound via Token-Bucket Proxy: max 40 req/min per merchant tenant.

9.7 AI-HUMAN SENTIMENT HANDOFF
  Run sentiment analysis on every inbound message (Tier 1, fast).
  
  Trigger "Priority Red" handoff if ANY of:
    - Sentiment score < -0.6
    - Phrases: "real person", "human", "manager", "refund", "lawyer",
               "this is ridiculous", "cancel", "lawsuit"
    - 3 consecutive messages with bot confidence < threshold
  
  On handoff:
    1. Pause bot immediately (conversation.status = 'HUMAN')
    2. Ping assigned human agent: in-app notification + email
    3. Display "Priority Red" badge in Inbox 2.0
    4. If no human claims within 10 minutes: send configured fallback
       message (e.g., "We'll get back to you within 2 hours + free code")
  
  Human toggle in Inbox 2.0: AI-Human Handoff toggle per conversation.
  When toggled ON (human mode): bot pauses, human types.
  When toggled OFF: bot resumes from where it left off.

9.8 FRICTION RECOVERY BLOCK
  This special flow node auto-activates when:
    - AI fails to understand user intent twice in a row, OR
    - Sentiment score drops below -0.7, OR
    - User explicitly expresses frustration (phrase detection)
  
  On activation:
    1. Immediately terminate bot's turn
    2. Fire "Priority Red" alert to human agent queue
    3. Send pre-configured "Friction Recovery" message:
       (customizable per brand: apology + compensatory offer)
    4. Set conversation.priority_red = TRUE

9.9 SHADOW A/B GHOST FLOW OPTIMIZER
  Every flow can have a Ghost Variant (flows.ghost_variant_id).
  Default ghost traffic: 10% (configurable).
  
  Optimizer tracks per variant: conversion rate, drop-off nodes, revenue.
  Auto-promote trigger: Ghost Variant outperforms main flow by >15%
  over 500+ interactions → surface "Promote Winner" prompt in UI.
  Demotion: underperforming Ghost Variant retired, traffic merged back.

9.10 LOYALTY HEAT MAPPING
  Signal sources per unified_contact_id:
    - IG/FB: posts liked, stories shared, DMs sent, Reels watched
    - TikTok: video watch duration (via deeplink pixel), comments, DMs
    - Purchases: from Shopify webhook or CAPI confirmation
  
  Loyalty score computation (0–100):
    Each signal type has a configurable weight.
    Score stored in unified_contacts.loyalty_score.
    Tier assignment:
      0–33: NEWBIE
      34–66: FAN
      67–100: ADVOCATE
  
  Warm hello injection: cross-platform context injected into AI prompt:
    "I noticed you checked out our Blue Hoodie on TikTok and liked

  Stored in unified_contacts.zero_party_signals JSONB.
